# Prompt: Build a Universal Genie Space Generator

You are building a **Databricks App** called `genie-space-generator` that provides a web UI where a salesperson can enter an industry, company name, and use case — and with one click generate a fully-configured Databricks Genie space backed by realistic synthetic data and metric views. This is a generalization of the `demand-forecasting-dbx-genie` reference implementation, which hardcodes a supply-chain/logistics scenario.

The app has two layers:
1. **A Gradio web frontend** — the salesperson-facing UI deployed as a Databricks App
2. **A Python core library** — the engine that generates domain models, synthetic data, metric views, and Genie spaces (reusable independently of the UI)

## Core Architecture Requirements

### 1. LLM-Powered Domain Generation (`generator.py`)

This is the key new module. It uses the Databricks Foundation Model API (via the MLflow deployments client) to generate the entire domain model. **This must run inside the Databricks notebook environment** using the workspace's built-in model serving.

The generator must produce a complete `DomainSpec` containing:

```python
@dataclass
class DomainSpec:
    """Complete domain specification generated by LLM."""

    # Identity
    company_name: str           # e.g., "MedFlow Analytics"
    industry: str               # e.g., "Healthcare"
    use_case: str               # e.g., "Patient readmission prediction..."
    space_title: str            # e.g., "MedFlow Analytics - Patient Readmissions & Capacity 🏥"
    space_description: str      # 2-3 sentence description for the Genie space
    schema_basename: str        # e.g., "medflow_readmissions"

    # Tables (list of TableSpec)
    tables: list[TableSpec]

    # Metric views (list of MetricViewSpec)
    metric_views: list[MetricViewSpec]

    # Genie configuration
    genie_instructions: str     # System prompt for the Genie assistant
    sample_questions: list[str] # 7 questions shown on landing page
    example_sqls: list[ExampleSQL]  # 7 question+SQL pairs for Genie training
    sql_snippets: SQLSnippets   # Filters, expressions, measures
    benchmarks: list[BenchmarkSpec]  # 7 benchmarks with expected SQL


@dataclass
class TableSpec:
    """Specification for a single synthetic table."""
    table_name: str             # e.g., "patient_encounters"
    description: str            # Table description for Genie
    columns: list[ColumnSpec]
    seasonal_patterns: dict     # Month-based volume patterns per category
    entity_dimension: str       # Which dimension fans out ("products", "locations", "time")

    # For deterministic SQL generation — the actual dimension data
    dimension_values: list[dict]  # Like PRODUCTS/WAREHOUSES in the reference
    category_distribution: dict   # Selection probability by category and month


@dataclass
class ColumnSpec:
    name: str
    sql_type: str               # e.g., "STRING", "INT", "DOUBLE", "DATE"
    comment: str                # Column comment for ALTER TABLE
    is_dimension: bool          # Whether to enable format_assistance + entity_matching in Genie
    generation_expr: str        # SQL expression for generating this column


@dataclass
class MetricViewSpec:
    """Specification for a YAML metric view."""
    view_name: str              # e.g., "patient_encounters_metrics"
    source_table: str           # Which table this metric view is built on
    dimensions: list[dict]      # [{name, expr}]
    measures: list[dict]        # [{name, expr, comment}]


@dataclass
class ExampleSQL:
    question: str
    sql_lines: list[str]


@dataclass
class SQLSnippets:
    filters: list[dict]         # [{sql, display_name, synonyms, instruction}]
    expressions: list[dict]     # [{alias, sql, display_name, synonyms}]
    measures: list[dict]        # [{alias, sql, display_name, synonyms}]


@dataclass
class BenchmarkSpec:
    question: str
    sql_lines: list[str]
```

### 2. Deterministic SQL Generation (`data.py`)

Once the `DomainSpec` is generated, the SQL builder must use **exactly the same deterministic patterns** as the reference implementation.

#### Hash-based determinism (replicate exactly)
```python
def _hash_fraction(seed: int, salt: str, *parts: str, scale: int = 10000) -> str:
    """Deterministic pseudo-random decimal in [0, 1)."""
    sql_parts = ", ".join([f"'{seed}'", f"'{salt}'", *parts])
    return f"(CAST(pmod(hash({sql_parts}), {scale}) AS DOUBLE) / {scale}.0)"

def _hash_int(seed: int, salt: str, *parts: str, modulo: int, offset: int = 0) -> str:
    """Deterministic pseudo-random integer."""
    sql_parts = ", ".join([f"'{seed}'", f"'{salt}'", *parts])
    base = f"pmod(hash({sql_parts}), {modulo})"
    if offset == 0: return base
    if offset > 0: return f"({base} + {offset})"
    return f"({base} - {abs(offset)})"
```

#### SQL generation pattern (replicate this structure for every table)

Each table's CTAS follows the reference pattern:
1. **VALUES CTEs** for dimension data (entities, locations) — rendered by `_values_sql()`
2. **Date range CTE** using `EXPLODE(SEQUENCE(DATE'{start_year}-01-01', DATE'2025-12-31', INTERVAL 1 DAY))` with scale-based start year (`2025 - (scale - 1)`)
3. **Cross-join skeleton** joining date_range × entities × locations, with hash-based noise columns
4. **Seasonal filtering** using category-month probability thresholds from `DomainSpec.tables[].seasonal_patterns` and `WHERE select_noise < selection_prob`
5. **Final SELECT** with category-based quantity ranges and status distributions

The key formula for row count targeting:
- `scale=1` → 1 year (2025), ~1,400 rows for transaction tables, ~4,400 for snapshot tables, ~870 for forecast tables
- Rows scale linearly with the `scale` parameter
- Weekday filtering (`WHERE dow BETWEEN 2 AND 6`) for transaction tables
- Weekly interval (`INTERVAL 7 DAY`) for snapshot tables
- Monthly interval (`INTERVAL 1 MONTH`) for forecast tables
- Product-warehouse pair filtering (`WHERE pair_hash < 52`) for snapshot/forecast tables to get ~52% of cross-join combinations

#### VALUES helper (replicate exactly)
```python
def _values_sql(rows: list[list[object]]) -> str:
    lines = []
    for idx, row in enumerate(rows):
        prefix = "    " if idx == 0 else "  , "
        rendered = ", ".join(_sql_value(value) for value in row)
        lines.append(f"{prefix}({rendered})")
    return "\n".join(lines)

def _sql_value(value: object) -> str:
    if isinstance(value, str):
        return "'" + value.replace("'", "''") + "'"
    return str(value)
```

## Critical Implementation Details

1. **The LLM call happens once** at the start of `deploy()`. Everything downstream is deterministic from the `DomainSpec` + `seed`.

2. **Dimension data is generated by the LLM, not SQL randomness.** The LLM produces realistic entity names, ID codes, facility names, region assignments, price/cost ranges, etc. These become VALUES clauses in the CTAS statements.

3. **Row generation uses the same hash-based determinism** as the reference. The LLM provides seasonal patterns and category distributions; the SQL builder applies them using `pmod(hash(...))`.

4. **No hardcoded domain knowledge** in any module except `generator.py`. The config, data, genie, validators, cleanup, and results modules must be fully generic and driven by the `DomainSpec`.

5. **The `scale` parameter** controls temporal range: `scale=1` = 2025 only, `scale=5` = 2021–2025. Start year = `2025 - (scale - 1)`.

6. **The three table archetypes** (transaction, snapshot, forecast) should be preserved as a pattern the LLM is guided to produce — these map naturally to most business domains.
